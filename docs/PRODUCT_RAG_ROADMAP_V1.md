# 产品级 RAG 完善路线图 v1

**读者**：项目 owner / 负责落地的工程师。  
**依据**：Sprint 5.3 已收尾（`docs/evals/sprint-5.3-closeout.md`、`evals/results/sprint-5.3-closeout-summary-001.md`）；**不**在本文件中重跑实验或改写 5.3 结论。

---

## 1. 当前状态总结

| 维度 | 事实摘要 |
|------|----------|
| **5.3 完成度** | 5.3a 合成 **12/12 pass**；5.3b **Paraphrase 18/18**、消融完成且 **Q1 归因 retrieval/ranking**，真实 PDF 摸底 **3/6**；5.3c 真实 PDF **5/6 pass**，**Q1/Q8** 改善，**Q9/Q11** 无谨慎壳退化。 |
| **当前最强能力** | 在**固定 gold** 与**规则抽检**下，**citation-first 回答 + 门控 + snippet 约束**与**多卷 PDF 上的分册偏置/注入/术语归一**已证明可把主链与关键定义题拉到可交付水平。 |
| **剩余主矛盾** | **跨分册、跨 chunk 边界的检索与排序治理**仍是系统级风险（5.3b 已用 Q1 证明）；真实 PDF 上 **Q6 partial** 为明确单点 follow-up（`docs/evals/sprint-5.3-q6-followup.md`）。 |
| **企业交付缺口** | 与 `AGENTS.md` / `TODO.md` 一致：除 RAG 质量外，**签名与公证、可重复 release、安装/升级/回滚、CI 发布链路**仍是产品级未完成项（见 `docs/RELEASE.md` 等现有文档）。 |

---

## 2. 下一阶段总目标（可执行、非愿景）

**在不大扩题集、不上复杂 judge、不大改 UI 的前提下：用小步真实问题回归锁住收益，并把 retrieval/ranking 从「技巧堆叠」升级为可维护的 query-type / source prior / metadata / chunk 边界治理，同时完成 macOS 发布级交付最小闭环。**

---

## 3. P0 / P1 / P2 规划

每项均含：**为什么现在做**、**目标**、**非目标**、**建议产出**、**验收方式**。

### P0 — 接下来最优先（必须先做）

#### P0-A 真实问题回归（小步扩展）

| 字段 | 内容 |
|------|------|
| **为什么现在做** | 5.3 已证明方法有效，但若只在 12 题合成 + 少量真实抽样上停留，**无法防止回归**；下一阶段必须把「高价值真实问法」**渐进**纳入 CI 或固定脚本，否则后续改检索易「看不见伤」。 |
| **目标** | 在**不一口气扩成大 benchmark** 的前提下，增加**少量**（例如每次 2～5 题）经产品/支持确认的高价值真实问题，与现有 gold/规则或轻量 checklist 绑定，可重复运行。 |
| **非目标** | 不扩成数十题竞赛集；不上 LLM judge；不为了覆盖率牺牲可解释失败原因。 |
| **建议产出** | 更新 `docs/EVAL_GUIDE.md` 或等价说明；新增或扩展 `evals/` 下**可脚本化**的真实/半真实回归入口；每次扩题附带「为何选这道题」。 |
| **验收方式** | CI 或本地一条命令可跑通；失败时日志/报告能定位到 **retrieval vs answer vs 规则** 之一；合并前对「新增题」有明确 owner 签字或 PR 说明。 |

#### P0-B retrieval / ranking 治理

| 字段 | 内容 |
|------|------|
| **为什么现在做** | 5.3b 消融明确 **Q1 在关 inject 时退化**；5.3c 用分册偏置、注入、chunk 合并等**修真实 PDF**，说明主矛盾是 **跨分册全流程主链召回** 与 **chunk/标题边界**，而不是再堆无结构的魔法系数。 |
| **目标** | 把现有有效手段**收束**为可文档化的策略：**query 类型**（全流程 / 定义 / 顺序 / 故障）→ **source prior（分册/元数据）** → **chunk 边界与合并规则** → 与 `searchChunks` 的接口清晰，便于评审与回滚。 |
| **非目标** | 不切换主架构为 GraphRAG；不大改 answer 主干；不引入需外部服务的复杂 rerank 平台。 |
| **建议产出** | 设计/文档节（可落在 `docs/`）；代码上**显式**的 query-type 或策略表（小模块）；对 P0-A 回归集的 **before/after** 报告模板。 |
| **验收方式** | 对固定回归集（含 5.3c 真实抽样脚本或等价）**无未解释的大幅退化**；Q1 类全流程题主 citation **不回落到明显无关分册**；变更附带简短「影响了哪些 query 类型」说明。 |

#### P0-C 发布级交付收口（macOS）

| 字段 | 内容 |
|------|------|
| **为什么现在做** | 桌面企业交付的硬门槛是**可安装、可升级、可验证来源**；RAG 再好也无法替代 **签名 / notarization / stapling** 与 **版本化产物**。与 `TODO.md` P0 安全/release 方向一致。 |
| **目标** | 明确并可重复：**构建产物命名与版本**、**签名与公证流程**（含失败排障）、**安装 / 升级 / 回滚**用户可见路径、**CI 或文档化的一键发布**（可先 semi-manual）。 |
| **非目标** | 不做多平台；不把发布系统做成复杂 SaaS；首轮不要求完美自动化，要求**可重复与可审计**。 |
| **建议产出** | 更新并执行 `docs/RELEASE.md` 中的缺口；`package.json`/CI 中与版本、artifact 一致的条目；必要时 support runbook 片段。 |
| **验收方式** | 至少在**测试证书/团队流程**下跑通一次完整链（或 staging 等价）；另一人仅按文档可复现核心步骤；版本号与下载产物一致可核对。 |

### P1 — 中优先级（产品稳定后继续补）

#### P1-1 Library 生命周期与运维叙事

| 字段 | 内容 |
|------|------|
| **为什么现在做** | import/reindex/repair 与 **library health** 完整化后，支持成本下降，且为检索质量提供稳定数据面。 |
| **目标** | 用户可理解的状态机：导入失败怎么办、如何重索引、如何导出诊断、何时需要重建索引。 |
| **非目标** | 不做多租户库管理；不做云同步优先。 |
| **建议产出** | Settings/文档中的运维路径；与健康检查联动的错误码或导出字段。 |
| **验收方式** | 模拟常见故障可恢复；支持人员可按文档引导用户。 |

#### P1-2 Support bundle 与流程标准化

| 字段 | 内容 |
|------|------|
| **为什么现在做** | 与 `AGENTS.md` Sprint 2 方向一致；企业场景下**可导出、可脱敏、不含原文默认**的支持包是交付标配。 |
| **目标** | 一键导出 support bundle；默认隐私边界清晰；与 runbook 对齐。 |
| **非目标** | 不做云端工单系统。 |
| **建议产出** | `docs/SUPPORT_RUNBOOK.md` 与 UI 导出入口；bundle 格式版本号。 |
| **验收方式** | 支持人员用 bundle 能回答「版本/库状态/最近错误」类问题。 |

#### P1-3 Answer-level eval（在 retrieval 之外看最终行为）

| 字段 | 内容 |
|------|------|
| **为什么现在做** | 仅有 chunk 级或关键词规则会漏「答非所问」；需在**可控成本**下看端到端 direct + citations。 |
| **目标** | 在**不默认上 LLM judge** 的前提下，扩展 checklist / 结构化字段比对（可复用 5.3 规则思路）。 |
| **非目标** | 不建设大规模自动评分平台。 |
| **建议产出** | eval 报告字段定义；可选「人工一票」流程。 |
| **验收方式** | 对固定集有可读 diff 报告。 |

#### P1-4 Embedding 可用性与失败策略（产品策略）

| 字段 | 内容 |
|------|------|
| **为什么现在做** | 向量缺失或模型加载失败时，用户应得到**确定行为**（降级纯 lexical、明确提示、可诊断），而非静默劣化。 |
| **目标** | 文档化 + 代码路径一致：何时 embed、失败如何记录、如何对用户说明。 |
| **非目标** | 不强制上云端 embedding。 |
| **建议产出** | 行为矩阵（有/无向量/磁盘满等）；日志字段。 |
| **验收方式** | 故障注入测试或通过文档演练可验证。 |

---

### P2 — 低优先级（当前不抢主线）

- **复杂 judge 系统**（多模型、高成本自动裁判）。
- **大规模 UI 改版**（除非为可访问性或合规所迫）。
- **重 agent 化**、多智能体编排优先于可靠检索与引用。
- **与当前「本地优先、可支持、可发布」无直接关系的能力外延**（如泛 GraphRAG 主链、多租户门户等）。

---

## 4. 推荐的阶段拆分（有阶段感、编号可调整）

| 阶段 | 侧重 | 说明 |
|------|------|------|
| **5.4（或下一命名）** | **P0-A + P0-B** | 真实回归小步扩容 + retrieval/ranking 治理文档与实现收束；**Q6 follow-up** 可插队为小任务，不升格为新大 sprint。 |
| **5.4.x / 并行轨道** | **P0-C** | 发布级交付可与 5.4 并行由专人推进，但**必须在里程碑上可见**，避免永远「功能先行、发布靠后」。 |
| **后续** | **P1** | 在 P0 无 open 风险项或已降级为已知技术债后批量推进。 |

---

## 5. 明确不做（与仓库边界一致）

- 不把主方向改成 **GraphRAG-first**、**agent-first**、**SaaS-first**。
- **不立刻**上复杂 **LLM judge** 作为合并门槛。
- **不做**大 UI 改版作为下一阶段主线。
- **不做**复杂 **agent orchestration** 替代 citation-first 与可诊断检索。
- **不**在未固化 P0-B 与 P0-A 的情况下继续**大幅扩 benchmark**（与 5.3 收口原则一致）。
