### B. 当前 chunk 行为（证据：真实跑分产物）

证据来源：`evals/results/sprint-5.3c-realpdf-run-001.json` 中 Q8 记录（真实 PDF 抽样 run）。

1) **定义片段落在一个“术语/表格混合 chunk”里**，且同 chunk 同时包含多条无关术语定义，说明边界不够“术语条目敏感”：

- citation snippet（节选，来自同一 chunk）包含：
  - `比值控制系统 ...`
  - `参数 数据逻辑组的名称...`
  - `参数对齐 该属性设为TRUE的参数... 如果为FALSE...`
  - `超时报警 ...`

这意味着：即便 `参数对齐` 的核心定义没有被切到别的 chunk，它也**经常被包裹在一个大杂烩 chunk**里，排序阶段会同时面对大量“术语列表/表格行”噪声块竞争。

2) **存在明显“表格行”型 sectionTitle 进入候选并给出很高 lexical 分，但质量分为负**：

- Q8 的 topResults 中（同 run 记录）：
  - `sectionTitle: "0.00 否 否 否 否 请赋值为“副调点名.OVE”"`（质量分为负）
  - `sectionTitle: "6.2.6.2 引脚"`（质量分为负）

这类候选通常来自：表格/列表被切成短行 + 断行拼接，导致**表头/数据行**变成可被 token 命中的“伪段落”。

### C. 为什么 B2/B3/B5 仍然不够

- **不是 B2 source prior 问题**：Q8 主 citation 已落到正确分册（手册 7），分册不再抢占。
- **不是 B1 queryType 问题**：Q8 属于 definition 类，路由已稳定。
- **不是 B3 haystack 问题**：标题/路径已进入 lexical haystack；Q8 本身不是“标题-only 关键词缺失”。
- **不是 B5 术语归一问题**：B5 解决的是 TRUE/FALSE、参数对齐相关词的“字面不稳”；但这里的主要风险在于：
  - chunk 中混入大量术语条目、表格行，导致候选集噪声大且可被词面命中；
  - 若关键条目与解释跨边界（或解释被短行切碎），仅靠 token 扩展无法保证“关键子串同 chunk”。
- **不是 answer 层问题**：该类失败首先表现为 retrieval/ranking 的证据粒度不稳定（证据段落本身被切碎/混入噪声），并非模板生成导致。

### D. 结论：是否值得开正式 B4

**结论：值得开正式 B4（但应保持最小规则、用单点用例验证）**。

理由（满足“值得开 B4”的门槛）：

- **稳定复现的风险形态明确**：术语/表格类内容在 chunk 层呈现“杂糅 + 噪声候选上位”的结构性风险。
- **归因清楚**：与分册 prior、query type、术语归一、answer 模板无关，核心在 **chunk boundary 对术语表/表格的敏感性不足**。
- **可用 1 条最小规则尝试改善**：不需要改 parsePdf 主链，只需在 `chunkText.ts` 引入一个“术语表/表格短行密度”保护策略即可做 before/after 验证。

## 极小预案（仅 1 条规则候选，供下一轮 B4）

### 建议的最小规则（只选 1 条）

**规则候选：术语/表格短行密度保护（PDF）**

- **触发条件**（保守，白名单信号）：
  - chunk 所在 sectionTitle/headingTrail 命中 `术语|参数|属性|引脚`；或
  - chunk 文本中出现 `TRUE|FALSE` 且同时出现 `该属性设为`（参数定义句式）；
  - 且 chunk 内存在“表格短行密度”信号（例如连续多行很短且包含 `是/否/TRUE/FALSE/数字` 的列状文本）。
- **动作**（保守粘合，避免切碎）：
  - 在 `chunkText.ts` 的切分阶段，对满足触发条件的区域：减少在短行之间的硬切分，优先把“术语名行 + 解释行（直到句号/空行/下一术语名）”粘合为同一 chunk。

### 本轮采用的单规则（已实现）

- **实现位置**：`src/lib/modules/chunk/chunkText.ts`
- **规则名（描述性）**：PDF `参数对齐` 术语/表格空行折叠（单窗口、单触发）
- **触发（白名单）**：
  - 文本出现 `参数对齐`，或出现 `该属性设为` 且同时出现 `TRUE` 与 `FALSE`
  - 仅在 PDF 路径启用（`pageSpans` 存在）
- **动作**：
  - 在触发点后有限窗口内，对“表格/短行”之间的空行做折叠（避免被 `\\n{2,}` 识别成 block 边界）
- **验收**：见 `chunkText.test.ts` 中 “B4 single-rule” 用例，检查关键子串是否落在同一 chunk（before/after）。

### 预期收益

- 让 `参数对齐 ... TRUE/FALSE ... 在线/离线/同步提示` 的关键子串更稳定落在**同一 chunk**，同时降低“表格行伪段落”进入 top-k 的概率。

### 预期风险

- chunk 可能变大，影响 embedding 成本与检索速度；需在规则内设置上限（例如触发区最多扩展 N 行或 N 字符）。
- 过度粘合可能把多个术语条目混在一起（但现状已存在），需要在验证中比较“关键子串集中度”和“噪声候选数量”。

### 建议改动文件

- `src/lib/modules/chunk/chunkText.ts`（B4 正式实现时）
- 增量单测：新增 1 个 fixture（从真实片段脱敏/截取文本）或直接写字符串输入，验证“关键子串同 chunk”。

### 如何验证（before/after）

- 输入：从真实 PDF 片段抽取的文本（包含 `参数对齐` 定义与若干表格/术语短行）。
- Before：现有 `chunkText` 输出中，“参数对齐 + TRUE + FALSE + 同步提示”是否常跨 chunk 或被夹杂到表格行 chunk 中。
- After：应用规则后，上述关键子串落在同一 chunk，且“表格行型 sectionTitle/短行 chunk”数量下降或质量分改善。
