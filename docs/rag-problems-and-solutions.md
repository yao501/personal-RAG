# Personal Knowledge RAG 项目中的典型问题与解法

这份文档整理了在实现本地 RAG 桌面应用过程中，真实遇到的一些效果问题，以及对应的分析思路和改进方案。

适合用于：

- 面试时讲项目难点
- 讲 RAG 调优经验
- 说明自己如何从“能跑”推进到“效果更可用”

## 项目背景

这个项目是一个本地优先的桌面 RAG 应用，目标是：

- 导入本地文档
- 建立索引
- 进行问答
- 输出明确 citation
- 让用户可以检查原始证据

支持文件类型包括：

- PDF
- Markdown
- TXT
- DOCX

在 MVP 跑通之后，真正的挑战不是“有没有答案”，而是：

- 检索是否够准
- chunk 是否切得合理
- 回答是否像回答，而不是摘抄
- citation 是否真的可检查、可理解

下面是这次项目中几个比较适合拿来讲的 RAG 相关问题。

---

## 1. 问题：初版回答过于“摘抄式”，不像真正的答案

### 现象

早期版本虽然能检索出相关 chunk，也能返回 citation，但回答整体比较像“把命中的文本拼起来”，而不是先回答问题。

典型表现：

- 回答很像原文摘录
- 可读性一般
- 用户需要自己从证据中再提炼结论

### 原因

初版 answer generation 逻辑过于直接：

- 取 top chunks
- 把其中的句子直接拼成回答

这样做的优点是简单、可控，但缺点也明显：

- 不能很好地区分“结论”和“证据”
- 很容易让回答显得啰嗦或机械
- 用户体验更像搜索结果，而不是问答

### 解法

把回答结构拆成三层：

1. `Direct Answer`
2. `Key Supporting Points`
3. `Citations / Source passages`

对应做法：

- 先生成一个更短、更直接的结论
- 再提炼 2 到 3 条 supporting points
- 最后把 citation 单独放到右侧，让证据和结论分层

### 结果

回答体验明显改善：

- 用户先看到结论
- 再看到支撑依据
- 最后再检查 citation

面试时可以总结成一句：

> 我把 RAG 的输出从“检索结果拼接”改成了“结论 + 支撑点 + 可检查证据”的三层结构，让它更像问答产品，而不是文档搜索。

---

## 2. 问题：检索相关性不足，过于依赖关键词

### 现象

初版检索主要基于词项匹配和简单词频打分。

存在的问题：

- 如果用户换一种说法提问，召回质量会明显下降
- 相似语义但不同表述的内容不容易被召回
- 排序更偏向词面匹配，而不是真正的语义相关性

### 原因

只有 lexical retrieval 时，系统对“语义近似”的理解能力有限。

也就是说：

- 适合精准词命中
- 不适合表达变体、概念转述、上下文相近的问题

### 解法

把检索改成混合检索：

1. 词法检索
2. metadata 命中加权
3. 轻量语义相似度
4. 本地 embedding 相似度
5. rerank

项目中具体做法：

- 保留关键词 / phrase 匹配
- 保留标题、section、path 的 metadata boost
- 增加轻量的字符级与 token overlap 相似度
- 接入本地 embedding 检索作为增强层
- 对 top candidates 再做一次 rerank

### 结果

效果上的改善主要体现在：

- 对用户提问方式更不敏感
- 相似语义的 chunk 更容易被召回
- 排序结果更稳定

面试时可以这样表达：

> 我没有停留在 BM25/关键词匹配，而是把检索升级成了 hybrid retrieval，并加入本地 embedding 和 rerank，使召回和排序都更贴近语义相关性。

---

## 3. 问题：最新内容没有被优先命中

### 现象

当知识库里存在多份主题相近、但更新时间不同的文档时，系统可能会命中旧内容，而不是用户更关心的“最新版本”。

例如：

- 用户问“最新状态是什么”
- 系统却把旧笔记排到前面

### 原因

单纯按 relevance 排序时，系统不知道“最近”这个维度对某些问题很重要。

### 解法

在排序里加入 freshness 维度，并识别 recency intent。

做法包括：

- 保存文档时间信息
  - importedAt
  - sourceUpdatedAt
  - sourceCreatedAt
- 当 query 包含 `latest`、`recent`、`current` 等意图时，提高 freshness 权重
- 在 relevance + semantic + rerank 基础上，再加入 freshness score

### 结果

系统在处理“最新/最近/当前”类问题时，更容易把更新的文档排到前面。

面试表述可以是：

> 我把检索排序从单纯 relevance 升级成了 relevance + freshness 的混合排序，特别是在带 recency intent 的查询上，会主动提升新文档优先级。

---

## 4. 问题：chunk 切片过于机械，语义边界不自然

### 现象

初版 chunking 比较接近“按 token 长度切块”，虽然能工作，但效果一般。

典型问题：

- 容易把段落中间切开
- 容易把标题和正文分裂
- 有些 chunk 看起来像半段话
- citation 读起来不自然

### 原因

单纯按长度切片很难保留文档结构。

特别是在 Markdown 或带标题层级的文档中：

- heading
- section
- paragraph

这些结构信息对检索质量非常重要。

### 解法

把 chunking 改成“结构优先”的自然切片。

具体做法：

- 优先识别 Markdown heading
- 保留 section title / section path
- 先按段落切
- 超长段落再按句子切
- 加 overlap 保留上下文衔接
- 避免过短引子单独成为一个 chunk

### 结果

chunk 更像完整语义单元，而不是机械片段。

直接收益包括：

- citation 更容易读
- supporting points 更稳定
- rerank 和 answer synthesis 更容易做

面试可讲：

> 我把 chunking 从长度驱动改成了结构驱动，优先保留 heading、section、paragraph 这些自然边界，从而提升了检索和 citation 的可读性。

## 5. 问题：supporting points 出现半截句、错误编号拼接

### 现象

在某些中文文档里，`Key Supporting Points` 会出现类似下面的问题：

- 半句话
- `而是：1.` 这种错误拼接
- 标题碎片或列表编号被直接抽出来

### 原因

早期 supporting point 抽取逻辑过于简单：

- 先把文本按换行压平
- 再按句号切句

这会把原文中的：

- 标题
- 列表项
- 下一行编号

错误地拼在一起。

### 解法

把 supporting point 提炼规则改得更保守。

做法包括：

- 先按行切，再按句切
- 过滤孤立编号、标题碎片、残句
- 过滤类似 `而是： 1.` 的 continuation-bad-case
- 如果抽不出足够好的句子，就回退成基于 citation 的稳定表述

### 结果

`Key Supporting Points` 从“能展示”提升到了“更可信、更像人在总结”。

这是一个很适合面试讲的点，因为它能体现：

- 不是所有 RAG 问题都在 retrieval
- synthesis / presentation 层也会显著影响产品体验

面试表述：

> 我发现用户感知到的“回答质量差”，有时并不是检索错了，而是 supporting point 的句子提炼出了半截句。后来我专门优化了这层抽取逻辑，并加回归测试锁住这类问题。

## 6. 问题：citation 看起来像“内容被截断”

### 现象

即使检索命中了正确 chunk，citation 卡片看起来也可能像是：

- 只有前半句
- 只显示到“更关键的是：”
- 看起来像 chunk 没切完整

### 原因

这里往往有两层原因：

1. chunk 边界刚好切在引子句附近
2. citation UI 默认只展示短 snippet

这两件事叠加后，用户会误以为“RAG 把内容切丢了”。

### 解法

做了两个方向的修复：

#### 一是优化 chunk 边界

- 避免在“更关键的是：”“如下：”“本质是：”这类 continuation marker 后直接断开
- 如果当前块是短引子，尽量把下一段正文一起带上

#### 二是优化 citation 展示

- 默认显示高质量 snippet
- 支持 `Show full chunk`
- 让用户可以直接展开查看完整 chunk

### 结果

citation 更可解释，也更容易做证据核查。

面试表达：

> 我发现 citation 的产品体验不仅取决于检索本身，还取决于 chunk 边界和 snippet 展示策略，所以我把“检索命中正确”进一步推进到了“用户一眼就能看懂证据”。

## 7. 问题：已有旧索引无法自动吃到新的检索能力

### 现象

项目在升级 chunking、metadata、embedding 后，旧文档如果不重新处理，就还停留在老的索引状态。

这会导致：

- 新逻辑写了，但旧库效果没变
- 用户以为“优化没生效”

### 原因

RAG 系统不是只改代码就行，索引本身也是数据资产。

### 解法

增加两个策略：

1. `Reindex` 显式重建索引
2. 对缺少 embedding 的旧 chunk 做自动补齐

### 结果

旧知识库也能逐步升级到新检索能力，不会因为历史数据导致“代码升级但效果没变化”。

面试可讲：

> 我把索引升级问题当成数据迁移问题来处理，而不是单纯代码问题，所以增加了 reindex 和 embedding backfill 机制，保证老数据也能享受到新的检索能力。

---

## 8. 问题：前端展示语言和产品定位不统一

### 现象

项目本身面向中文场景，但前端界面最初是中英混合的：

- `Library`
- `Chat`
- `Settings`
- `Direct Answer`

这会带来两个问题：

- 产品感不统一
- 用户理解成本更高

### 解法

把用户可见文案统一中文化：

- `资料库`
- `问答`
- `设置`
- `直接答案`
- `关键支撑点`
- `来源片段`

### 结果

整个产品的定位和体验更统一。

这个点虽然看似不是核心算法问题，但在面试里能体现你对“算法效果”和“产品落地体验”是一起考虑的。

---

## 一句话总结：这个项目里的 RAG 调优思路

如果面试官问你“你在这个项目里做了哪些 RAG 优化”，可以归纳成下面这套结构：

### 1. 检索层

- 从纯关键词匹配升级成 hybrid retrieval
- 加入 metadata、embedding、rerank、freshness

### 2. 切片层

- 从机械切片升级成结构化自然切片
- 保留标题、section、上下文

### 3. 回答层

- 从摘抄式输出升级成“直接答案 + 支撑点 + citation”

### 4. 证据层

- 优化 citation 可读性
- 支持展开完整 chunk
- 保持答案 grounded 且可检查

### 5. 工程层

- 支持 reindex
- 支持旧索引升级
- 支持本地 embedding 增强

## 面试时可以直接使用的表达模板

你可以这样讲：

> 这个项目一开始的 MVP 重点是先把本地导入、索引、问答和 citation 跑通，但跑通之后我发现真正决定体验的不是“能不能回答”，而是回答是不是像答案、检索是不是够准、citation 是不是一眼能看懂。  
>  
> 所以后面我主要做了几类优化：  
> 第一类是 retrieval，从纯词法检索升级成 hybrid retrieval，加入 metadata、embedding、rerank 和 freshness；  
> 第二类是 chunking，从按长度切块改成按 heading/section/paragraph 的自然边界切片；  
> 第三类是 answer generation，把输出改成 direct answer + supporting points + citations；  
> 第四类是 citation 可解释性，支持展开完整 chunk，避免用户觉得证据被截断。  
>  
> 这些优化的核心不是单点算法，而是把 RAG 当成一个完整系统去调：检索、切片、生成、展示和索引迁移都要一起考虑。
